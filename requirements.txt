Terraform-инфраструктура для кластера

Кластер Kubernetes
Object Storage
Postgresql

Требования
Конфигурация Terraform должна находиться в директории terraform.

Задачи
Установите YC CLI и выполните аутентификацию с первичной настройкой утилиты.
Создайте в директории terraform новый проект, подключите провайдер Yandex и настройте хранение бекенда в S3.
Опишите VPC с подсетями, NAT и security group, чтобы обеспечить сеть для кластера.
Создайте ресурс Managed Kubernetes с привязкой к подготовленной сети и сервисным аккаунтам.
Настройте группу нод с требуемым типом и количеством инстансов для worker-пула.
Разверните управляемый PostgreSQL (или другую выбранную СУБД), определите security group и параметры доступа, совместимые с кластером.
Создайте Object Storage bucket и доступы (access/secret key) для приложения.
Создайте Lockbox‑секрет для хранения чувствительных параметров приложения (DB/S3), зафиксируйте его ID.
Сформируйте outputs с kubeconfig/token, строкой подключения к БД, доступами S3 и ID Lockbox.
Опишите в Makefile команды для работы с Terraform.
В README.md добавьте требования к рабочей системе (с которой будет выполняться создание инфраструктуры), примеры команд.
Подсказки
Не храните состояние Terraform в репозитории.
Для работы Terraform с Яндекс можно использовать переменные окружения или запросить токен командой:
yc iam create-token
Итог шага
В репозитории есть Terraform-код сети и кластера, поддерживающий init/plan/apply без ручных правок.
Managed Kubernetes кластер с одной нодой успешно создан и доступен через kubeconfig.
Управляемая база данных запущена, доступна из кластера и задокументированы параметры подключения.
Outputs содержат ID кластера, группы нод, kubeconfig/token, параметры БД, доступы S3 и ID Lockbox для следующих шагов.
Terraform-конфигурация лежит в terraform/ (например, main.tf, variables.tf, outputs.tf, providers.tf, versions.tf).

Kubernetes-манифесты и первичный деплой приложения
На этом шаге опишем базовые ресурсы в кластере и подготовим приложение к деплою.

Требования
Манифесты kubernetes должны находиться в директории k8s
Ссылки
Managed Service for Kubernetes
Kubernetes Deployments
Probes и стратегии
Секреты и ConfigMap
Шпаргалка по Kubernetes
Задачи
Создайте отдельный namespace приложения и опишите для него ConfigMap с конфигурацией и Secret с чувствительными данными, чтобы контейнеры получали все параметры из кластера.
Опишите Deployment для приложения с RollingUpdate, лимитами CPU/RAM и probes (readiness/liveness), чтобы обновления шли без даунтайма и платформа контролировала жизнеспособность.
Настройте Service для внутреннего трафика и удобной проверки результата (например, через port-forward).
Примените манифесты и через kubectl подтвердите состояние Ready у подов и успешные ответы сервиса.
Обновите README.md и добавьте в Makefile необходимые команды.
Итог шага
В репозитории есть каталог k8s/ с namespace, ConfigMap, Secret и Deployment/Service.
Приложение успешно запущено в кластере, сервис отвечает внутри кластера/через временный доступ.
Документирован базовый процесс выката (kubectl apply), готов к автоматизации.

Масштабирование, балансировка и zero-downtime релизы
Расширяем кластер до нескольких нод, подключаем балансировщик и гарантируем безостановочные релизы.

Ссылки
Node groups и autoscaling
PodDisruptionBudget
Ingress NGINX
Задачи
Увеличьте количество рабочих нод через Terraform, чтобы кластер располагал минимум двумя инстансами.
Обеспечьте внешний доступ к приложению: либо Ingress + ingress‑controller, либо Service типа LoadBalancer с публичным IP.
Настройте стратегию обновления, PodDisruptionBudget и (опционально) HPA для безостановочных релизов.
Проведите rolling update новой версии образа и подтвердите равномерное распределение трафика без ошибок 5xx (например, через несколько запросов подряд).
Итог шага
Кластер масштабирован до минимум двух рабочих нод.
Настроен внешний доступ к приложению (Ingress или Service LoadBalancer).
Rolling update проверен: релизы проходят без простоя, ресурсы PDB/HPA задокументированы.
Изменения для масштабирования нод лежат в terraform/.
Манифесты балансировки и масштабирования лежат в k8s/.

Управление секретами через Yandex Lockbox
Интегрируем Lockbox с Kubernetes, чтобы секреты обновлялись автоматически и не попадали в репозиторий.

Ссылки
Начало работы с Yandex Lockbox
Справочник Terraform для Yandex Lockbox
External Secrets Operator
Синхронизация с секретами Yandex Managed Service for Kubernetes
Задачи
Создайте Lockbox‑секрет для всех чувствительных данных проекта и зафиксируйте его ID.
Разверните интеграцию (External Secrets Operator) и обеспечьте сервисному аккаунту права на чтение Lockbox.
Настройте синхронизацию Lockbox значений в Kubernetes Secrets и переведите приложение на использование этих секретов (через Helm values).
Отработайте ротацию значения в Lockbox и подтвердите, что кластер автоматически получает обновленные данные без простоя.
Итог шага
Все чувствительные данные вынесены в Lockbox, нет секретов в Git.
Kubernetes автоматически подтягивает значения и обновляется при ротации.
Процесс ротации документирован и проверен на стенде.
